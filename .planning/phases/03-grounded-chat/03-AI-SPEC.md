# AI-SPEC — Phase 3: Grounded Chat

> AI design contract generated by `$gsd-ai-integration-phase`. Consumed by planning and evaluation.

## 1. System Classification

**System Type:** Conversational

**Description:** A civic assistant answers only from a small, server-derived packet of already surfaced public-record evidence for the active Lamplighter coverage. Good output is short, grounded, locally citation-validated, and clear when the records do not answer.

**Critical Failure Modes:**

1. A factual answer appears without an application-issued evidence ID.
2. A provider receives or returns evidence outside the selected location and surfaced packet.
3. A provider failure changes into an unsupported answer, or an invalid answer falls through to another provider.

## 1b. Domain Context

Lamplighter provides public civic information, not legal advice or an authoritative record of current law. A wrong or overconfident answer can misstate a public official, candidate, policy, or government action and harm public trust and reputation.

### Grounded-answer rubric ingredients

| Dimension | Pass | Fail |
|---|---|---|
| Citation precision | Every displayed factual block cites one or more issued IDs whose exact quoted evidence supports that block. | Citation is missing, points to another location/record, or does not support the claim. |
| Selected-coverage adherence | Uses only the active coverage packet; Indianapolis may include its configured state/federal records, while Indiana and federal remain exact-level. | Answers from another location, arbitrary web material, or un-surfaced records. |
| Refusal | Returns the fixed insufficient-evidence response when the packet does not directly answer. | Guesses, fills gaps from general knowledge, or treats a feed date as legal/policy status. |
| Source distinction | Identifies reporting as reporting; government records remain primary records and conflicting supported claims are presented without choosing a winner. | Presents reporting as an official decision, or erases a material supported disagreement. |
| Candidate positions | Attributes a position only to an explicit, recent public statement in the packet. | Infers a position from party, vote, affiliation, or silence. |

### Review lens

Have a civic-record reviewer (municipal clerk, legislative researcher, public-interest journalist, or government-affairs analyst) assess the small reference set against the stored records. The source of truth is the exact stored public quote, its canonical source URL, and its page/timestamp locator—not model knowledge, search results, or an interpretation beyond the record.

## 2. Framework Decision

**Selected Framework:** Direct official provider SDK calls (no agent framework)

**Version:** Current official SDK versions, verified during implementation.

**Rationale:** This is a linear closed-evidence Q&A flow. Direct adapters keep the GPT-5.6-first fallback order, application-issued citation IDs, and validation/error classification explicit without agent-framework state or retrieval abstraction.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|---|---|
| LangChain TypeScript | Adds abstraction before the three bounded adapters have demonstrated meaningful duplication. |

**Vendor Lock-In Accepted:** Partial — OpenAI is primary; Anthropic and Gemini are optional server-side fallbacks behind the same local answer contract.

## 3. Framework Quick Reference

This phase needs no agent framework: one server-only request produces one bounded JSON answer. Use the official OpenAI Node SDK for the primary call and small server-side `fetch` adapters for the optional Anthropic and Gemini fallbacks. Do not add a retrieval, agent, tracing, or provider-selection package.

| Piece | Responsibility |
|---|---|
| `buildPacket(coverageId)` | Re-derives the active coverage and six newest surfaced records; issues the allowed evidence IDs. |
| `answerWithOpenAI` | Calls `responses.create` with GPT-5.6 and strict JSON Schema. |
| `answerWithAnthropic` / `answerWithGemini` | Call their JSON-output endpoints only when their server-side key exists. |
| `parseAnswerBlock` | Validates the common answer shape and rejects IDs outside the packet. |
| bundled answers | Resolves known demo prompts without any network or provider key. |

The common provider output is deliberately small:

```ts
type AnswerBlock = { text: string; evidenceIds: string[] };
type GroundedAnswer =
  | { status: "answered"; blocks: AnswerBlock[] }
  | { status: "insufficient"; blocks: [] };
```

Use this JSON Schema for every provider, keeping it shallow because Gemini supports a JSON-Schema subset:

```ts
const answerSchema = {
  type: "object",
  additionalProperties: false,
  required: ["status", "blocks"],
  properties: {
    status: { type: "string", enum: ["answered", "insufficient"] },
    blocks: {
      type: "array", maxItems: 4,
      items: {
        type: "object", additionalProperties: false,
        required: ["text", "evidenceIds"],
        properties: {
          text: { type: "string", maxLength: 500 },
          evidenceIds: { type: "array", minItems: 1, items: { type: "string" } },
        },
      },
    },
  },
} as const;
```

OpenAI’s Responses API supports Structured Outputs; use `text.format` with `type: "json_schema"`, `strict: true`, and this schema. Anthropic JSON output uses `output_config.format` with `type: "json_schema"`; Gemini JSON output uses a JSON response format/schema. All three still require local semantic validation.

## 4. Implementation Guidance

Keep the endpoint as a closed-data transformation: accept only `{ coverageId, question }`, validate both, derive the packet on the server, then expose only its exact quotes and issued IDs to a configured provider. The browser never supplies evidence, record text, URLs, or provider choice.

```ts
const response = await openai.responses.create({
  model: process.env.OPENAI_MODEL ?? "gpt-5.6",
  input: [
    { role: "system", content: "Answer only from PACKET. Every answered block needs one or more PACKET evidence IDs. Return insufficient when it is not answered." },
    { role: "user", content: JSON.stringify({ question, packet }) },
  ],
  max_output_tokens: 450,
  text: { format: { type: "json_schema", name: "grounded_answer", strict: true, schema: answerSchema } },
});
const answer = parseAnswerBlock(JSON.parse(response.output_text), allowedEvidenceIds);
```

`parseAnswerBlock` is the trust boundary: require the exact two result variants; trim non-empty block text; require at least one evidence ID for every displayed block; require all IDs in the packet; reject an `answered` result with no blocks. It maps failure to the fixed insufficient-evidence result and never renders raw provider text.

Call configured adapters in the locked order OpenAI → Anthropic → Gemini. Classify only abort/timeout, network failure, HTTP 408, 429, and 5xx as transient. Move to the next adapter only for that class. Missing key, refusal/insufficient output, malformed JSON, schema/local-validation failure, unknown IDs, and other 4xx results are terminal; return the local refusal rather than trying another model. Use one short abort timeout per adapter and no automatic retry loop.

Use bundled preset answers before live calls when no live provider is configured. They use the same `GroundedAnswer` validation and packet IDs, so the citation renderer has one path. Do not send history to a provider in this phase; `localStorage` history is UI state only and threads remain separated by coverage ID.

## 4b. AI Systems Best Practices

### Structured outputs and local validation

Use the single JSON Schema above plus a tiny TypeScript runtime validator; do not add Zod solely for this one shallow shape. Provider-constrained JSON prevents syntax drift, while the application enforces the important semantics: issued IDs only, citation per displayed block, packet boundary, and the candidate-position rule. A schema-compliant but semantically unsupported answer is insufficient evidence, not a retry.

### Prompt and context discipline

Keep the system instruction stable and short: facts must come only from `PACKET`, each answer block must cite packet IDs, disagreements must present both claims, and unanswered questions must return `insufficient`. The user message contains the question plus no more than six deterministic records, each limited to title, date, type, source label, locator, exact quote, and issued ID. Do not enable web search, file search, tools, or arbitrary URLs.

### Latency, cost, and failure handling

Bound the answer at 450 output tokens and avoid streaming: the complete JSON must pass local validation before any text renders. Skip absent provider keys. Surface only the active provider for success, or the failed provider and compact error class for an unavailable live request. Bundled answers stay available without a key. No tracing, eval service, cache, conversation summarization, or background ingestion is needed for this small offline demo.

## 5. Evaluation Strategy

### Deterministic regression suite

Use one Node script, run alongside the existing seed and scope checks, with a fixed closed fixture of **12 cases** drawn from the bundled records. It exercises the common validator and provider-selection function with fake adapter results; it never calls a model or network service. The script must fail nonzero on any regression.

| Dimension | Cases | Pass condition |
|---|---:|---|
| Schema and citation integrity | 4 | Valid answer blocks retain only issued IDs; malformed payload, empty text/IDs, and an `answered` result without blocks become the fixed insufficient-evidence result. |
| Coverage isolation | 3 | Indianapolis packet contains only its configured Indianapolis/Indiana/federal records; Indiana and U.S. federal remain exact-level; a valid ID from another packet is rejected. |
| Refusal and civic attribution | 2 | An unsupported question and a candidate-position question without an explicit recent public statement both return insufficient evidence with no answer text. |
| Fallback classification | 3 | OpenAI timeout/429 tries the next configured adapter; a missing key is skipped; malformed output, unknown IDs, refusal/insufficient output, and non-transient 4xx stop the chain. |

Keep five or more normalized preset questions in the same fixture. Each must resolve to a packet-valid bundled answer for its coverage; an arbitrary offline question must refuse. This is the offline regression flywheel: add a minimal fixture row whenever a real demo failure is fixed, then keep it in the no-network script. Do not introduce Promptfoo, a judge model, or an evaluation service for this deadline demo.

### Human review before recording

For the five-plus bundled answers and any changed live prompt, compare each displayed block against its stored quote, locator, and source label. Pass only if every claim is directly supported, reporting remains labeled, and conflicts are presented without declaring a winner. A civic-record reviewer is the desired reviewer when available; otherwise the project owner performs this short recorded-data review. LLM-as-judge is deferred because it is neither calibrated nor needed to enforce the closed-packet contract.

## 6. Guardrails

The online guardrail is the server response validator, run after every bundled or live provider response and before any text reaches React:

1. Re-derive coverage and the capped packet from the server-side corpus; ignore client evidence, source, URL, and provider fields.
2. Accept only the exact `GroundedAnswer` variants. Every displayed block must have non-empty text and one or more issued evidence IDs from that packet.
3. Map malformed output, unknown/cross-coverage IDs, uncited blocks, an unsupported candidate-position claim, and terminal provider failures to the fixed insufficient-evidence response. Never render raw model text.
4. Only timeout, network, HTTP 408/429, and 5xx may advance GPT-5.6 → Anthropic → Gemini. Any grounding/validation failure is terminal and cannot cause provider fallback.

The UI then resolves IDs to its existing evidence panel; it never accepts provider-generated URLs, locators, or citation labels. `Find more sources` remains explanatory-only in this phase and must not fetch or widen the packet.

## 7. Production Monitoring

No telemetry, tracing backend, or persistent prompt logging is added for the offline, no-account demo. The only production-facing signal is the existing safe UI state: provider name on a successful live answer, or failed provider plus compact error class when live AI is unavailable. Never log questions, evidence quotes, API keys, or raw provider output to the browser.

Before deployment or recording, run the deterministic regression suite, existing evidence/seed/scope/import checks, and the production build. After the event, if live usage continues, add opt-in server-side aggregate counters for answer/refusal/validation-failure/fallback class and periodically sample with explicit retention rules; only then consider tracing or a calibrated human/LLM review loop.
