# Phase 3: Grounded Chat - Context

**Gathered:** 2026-07-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Turn Lamplighter’s existing location-scoped Recent evidence into a Chat view that answers only from a small closed packet of surfaced records, validates every returned evidence ID before display, and clearly refuses unsupported questions. The no-key demo remains usable through bundled cited answers; live answers are server-side and provider-backed.

</domain>

<decisions>
## Implementation Decisions

### Answer modes and providers

- **D-01:** Without a live key, show that Live AI is unavailable while keeping five or more clearly labeled, evidence-backed bundled demo questions usable.
- **D-02:** Label deterministic output `Bundled demo answer`; it must never claim to be generated by a live provider.
- **D-03:** Live answers use GPT-5.6 first, then Anthropic, then Gemini when their optional server-side keys are configured. Skip providers whose keys are absent.
- **D-04:** Fail over only after a provider outage, timeout, or rate limit. Never fail over after insufficient evidence, malformed output, unknown citations, or an uncited claim.
- **D-05:** Show the actual active provider on each live answer; keep live-provider failure details to provider name and error type.

### Chat scope and persistence

- **D-06:** Chat uses the active Recent coverage selector; changing coverage starts a distinct evidence scope.
- **D-07:** Keep conversation history in browser local storage, separated into visible threads by location coverage.
- **D-08:** Show five location-aware suggested questions above a free-text composer.
- **D-09:** A live answer visibly states the selected location and the count of surfaced records in its evidence packet.

### Citations, refusal, and disagreement

- **D-10:** Render citation chips after every factual answer sentence or bullet. Chips open the same provenance panel used by Recent.
- **D-11:** Use a short direct answer first, followed by evidence-backed bullets when useful.
- **D-12:** When evidence is insufficient, say: `I can’t answer that from the selected public records yet,` then offer relevant suggested questions.
- **D-13:** If records materially differ, state that sources differ, show each cited claim, and do not select a winner.
- **D-14:** Show a visibly scoped future `Find more sources` affordance on refusal, but it must not search, ingest, or substitute sources in this phase.

### Model and evidence safety

- **D-15:** Accept only structured live-provider answer blocks containing application-issued evidence IDs. Application code, never the provider, maps IDs to citation chips.
- **D-16:** Send only a small deterministic subset of the selected location’s already-surfaced, exact-quote-backed records to a provider.
- **D-17:** Treat malformed structured output, unknown evidence IDs, or factual content without citations as insufficient evidence; do not display the invalid model text.
- **D-18:** Limit live answer length only; do not add extra demo interaction caps.
- **D-19:** Answer a candidate or office-holder position only from an explicit recent public statement in the supplied evidence. Never infer it from party, vote, or affiliation.

### the agent's Discretion

- Choose the smallest server-only provider adapter and strict structured schema that support the locked GPT-5.6-first fallback order.
- Choose the deterministic evidence-packet size, answer-length cap, preset wording, and browser-storage structure without widening evidence or location access.
- Reuse the existing evidence-panel interaction and custom CSS system rather than add a UI library.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets

- `src/App.tsx`: native coverage select, deterministic eligible-record calculation, selected-record state, and evidence detail panel.
- `src/demo-seed.ts` and `src/types.ts`: static browser-safe closed corpus with canonical URLs, exact quotes, and locators.
- `scripts/evidence-contract.mjs`: deterministic validation helpers for provenance, source membership, and location boundaries.
- `src/styles.css`: responsive custom-CSS shell, panel behavior, and fixed bottom navigation.

### Established Patterns

- The Vite client reads static generated seed data only; it does not access SQLite, provider modules, raw documents, or secrets.
- Recent treats reporting as explicit reporting and never turns dates/classifications into legal or policy status.
- Invalid bundled data fails loudly instead of fetching, rebuilding, or broadening coverage.

### Integration Points

- Replace the unavailable Chat bottom-nav action with a Chat view while preserving Recent and its evidence panel.
- Add server-only provider/model handling outside the client bundle and validate output against the selected seed evidence IDs before React renders it.
- Reuse the current selected location coverage relation to form the closed evidence packet and to separate local threads.

</code_context>

<specifics>
## Specific Ideas

- The demo needs a reliable screen-recording path even without any provider key, while visibly distinguishing bundled answers from live-provider answers.
- The UI should disclose the live provider per answer and show the location/record count used, without exposing secrets or raw provider diagnostics.

</specifics>

<deferred>
## Deferred Ideas

- **Evidence expansion:** an explicit refresh/search workflow that searches configured government sources first, then clearly labeled reporting, validates and surfaces new records before Chat uses them. It must not be hidden inside an answer request.
- User-selectable provider/model choice — the fallback order is deployment-controlled in Phase 3.
- Any inference of candidate positions from party, voting patterns, or affiliation.

</deferred>
