<threat_model>
## Trust Boundaries

| Boundary | Description |
|---|---|
| Client request to Node route | A modified browser can submit forged evidence, provider choices, locations, or oversized input. |
| SQLite packet to provider | The provider must receive only configured, selected-location exact quotes and issued IDs. |
| Provider output to React | Well-formed JSON can still contain unsupported text or forged/cross-coverage citations. |
| Provider configuration to UI | Availability diagnostics must not disclose API keys, raw errors, prompts, or model output. |

## STRIDE Threat Register

| Threat ID | Category | Severity | Mitigation |
|---|---|---:|---|
| T-03-01 | Tampering | high | Ignore all client evidence/provider fields; server derives packet and allow-list from read-only SQLite. |
| T-03-02 | Spoofing | high | Accept only application-issued IDs and resolve chips locally; provider never supplies links/labels. |
| T-03-03 | Information disclosure | high | Node-only adapters, status allowlist, no prompt/output logging, no client key/config transfer. |
| T-03-04 | Elevation of privilege | high | No tools, URLs, fetch/search, import, or provider selection exposed through chat. |
| T-03-04a | Integrity | high | Candidate-position preflight requires `recent_public_position` inside the exact packet and refuses before model invocation when absent. |
| T-03-05 | Denial of service | medium | Ordinary transport/body safety, fixed packet/output caps, one timeout, no retry loop; no product question or interaction cap. |
| T-03-06 | Repudiation | medium | Deterministic packet/validation check and provider/error class are retained only in the response state. |
</threat_model>
