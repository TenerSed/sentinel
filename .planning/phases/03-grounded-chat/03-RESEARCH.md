# Phase 3: Grounded Chat — Research

## User Constraints

### Answer modes and providers

- **D-01:** Without a live key, show that Live AI is unavailable while keeping five or more clearly labeled, evidence-backed bundled demo questions usable.
- **D-02:** Label deterministic output `Bundled demo answer`; it must never claim to be generated by a live provider.
- **D-03:** Live answers use GPT-5.6 first, then Anthropic, then Gemini when their optional server-side keys are configured. Skip providers whose keys are absent.
- **D-04:** Fail over only after a provider outage, timeout, or rate limit. Never fail over after insufficient evidence, malformed output, unknown citations, or an uncited claim.
- **D-05:** Show the actual active provider on each live answer; keep live-provider failure details to provider name and error type.

### Chat scope and persistence

- **D-06:** Chat uses the active Recent coverage selector; changing coverage starts a distinct evidence scope.
- **D-07:** Keep conversation history in browser local storage, separated into visible threads by location coverage.
- **D-08:** Show five location-aware suggested questions above a free-text composer.
- **D-09:** A live answer visibly states the selected location and the count of surfaced records in its evidence packet.

### Citations, refusal, and disagreement

- **D-10:** Render citation chips after every factual answer sentence or bullet. Chips open the same provenance panel used by Recent.
- **D-11:** Use a short direct answer first, followed by evidence-backed bullets when useful.
- **D-12:** When evidence is insufficient, say: `I can’t answer that from the selected public records yet,` then offer relevant suggested questions.
- **D-13:** If records materially differ, state that sources differ, show each cited claim, and do not select a winner.
- **D-14:** Show a visibly scoped future `Find more sources` affordance on refusal, but it must not search, ingest, or substitute sources in this phase.

### Model and evidence safety

- **D-15:** Accept only structured live-provider answer blocks containing application-issued evidence IDs. Application code, never the provider, maps IDs to citation chips.
- **D-16:** Send only a small deterministic subset of the selected location’s already-surfaced, exact-quote-backed records to a provider.
- **D-17:** Treat malformed structured output, unknown evidence IDs, or factual content without citations as insufficient evidence; do not display the invalid model text.
- **D-18:** Limit live answer length only; do not add extra demo interaction caps.
- **D-19:** Answer a candidate or office-holder position only from an explicit recent public statement in the supplied evidence. Never infer it from party, vote, or affiliation.

### the agent's Discretion

- Choose the smallest server-only provider adapter and strict structured schema that support the locked GPT-5.6-first fallback order.
- Choose the deterministic evidence-packet size, answer-length cap, preset wording, and browser-storage structure without widening evidence or location access.
- Reuse the existing evidence-panel interaction and custom CSS system rather than add a UI library.

### Deferred Ideas

- **Evidence expansion:** an explicit refresh/search workflow that searches configured government sources first, then clearly labeled reporting, validates and surfaces new records before Chat uses them. It must not be hidden inside an answer request.
- User-selectable provider/model choice — the fallback order is deployment-controlled in Phase 3.
- Any inference of candidate positions from party, voting patterns, or affiliation.

## Architecture Patterns

### 1. One answer contract for bundled and live output

Use one browser-safe result shape, such as:

```ts
type AnswerBlock = { text: string; evidenceIds: string[] };
type GroundedAnswer = {
  status: "answered" | "insufficient";
  blocks: AnswerBlock[];
};
```

For `answered`, require at least one non-empty `evidenceIds` array on **every** block. For `insufficient`, require no answer blocks. Render one block as the direct answer and remaining blocks as bullets; each block’s IDs become chips after that sentence/bullet. [VERIFIED: 03-CONTEXT.md]

Do not accept model-generated URLs, titles, locators, source labels, HTML, Markdown citation syntax, or arbitrary citation strings. Resolve IDs only from the closed server-generated map. [VERIFIED: 03-CONTEXT.md]

### 2. Re-derive the closed packet on the server

`POST /api/chat` should accept `{ coverageId, question }`; validate the coverage ID, calculate covered locations using the same relation as Recent, and select a deterministic cap (recommend six newest eligible records, ordered by existing `publishedAt DESC, id ASC`). Send only each record’s application-issued ID, title, update type, publisher/source kind, date, exact quote, and locator. [VERIFIED: src/App.tsx, src/demo-seed.ts, scripts/recent-scope-check.mjs]

The server should ignore any client-supplied evidence IDs or record content. It must make the exact allowed-ID set before a provider call and validate returned IDs against that set after the call. [VERIFIED: 03-CONTEXT.md]

### 3. Explicit provider result classification

Each provider adapter returns exactly one of: `success`, `transient_failure`, or `terminal_failure`. Continue GPT-5.6 → Anthropic → Gemini only for an abort/timeout, network outage, HTTP 408, HTTP 429, or HTTP 5xx. Treat malformed JSON, schema mismatch, a refusal/insufficient response, unknown evidence IDs, validation failure, missing key, and other 4xx responses as terminal. [VERIFIED: 03-CONTEXT.md] [CITED: https://developers.openai.com/api/docs/guides/error-codes] [CITED: https://platform.claude.com/docs/en/api/errors] [CITED: https://ai.google.dev/gemini-api/docs/troubleshooting]

Use a single short `AbortSignal.timeout(...)` per call and no automatic retry loop in Lamplighter. This keeps the fallback order observable and prevents a provider SDK’s retry behavior from obscuring a failover. Anthropic’s SDK otherwise retries transient failures twice by default. [CITED: https://platform.claude.com/docs/en/api/errors]

### 4. Deterministic demo before live AI

Match five or more normalized, location-aware preset questions to pre-authored `GroundedAnswer` records built from the existing IDs. Any unrecognized offline question returns the exact insufficient-evidence copy. A configured provider is only used for free-text/live questions; the UI still states `Live AI unavailable` when no usable key exists. [VERIFIED: 03-CONTEXT.md]

Store conversation threads by coverage ID in `localStorage` with a versioned key and validate parsed browser data before rendering. Clear neither prior coverage’s thread nor evidence scope: switching coverage shows a separate thread. [VERIFIED: 03-CONTEXT.md]

## Don’t Hand-Roll

- Do not build RAG, embeddings, FTS, a vector database, an agent loop, or a web-search tool: Phase 3’s corpus is already surfaced, bounded, and deliberately closed. [VERIFIED: 03-CONTEXT.md]
- Do not build citation parsing from free-form prose: use structured `evidenceIds` and resolve them through the server map. [VERIFIED: 03-CONTEXT.md]
- Do not send conversation history, browser storage, raw documents, API keys, or client-selected sources to providers. Only the question plus closed evidence packet belongs in the request. [VERIFIED: AGENTS.md, 03-CONTEXT.md]
- Do not create a shared provider framework or provider selector. Three direct functions behind one narrow result union are enough for this deadline phase. [VERIFIED: 03-CONTEXT.md]

## Common Pitfalls

1. **Trusting the static client seed as an authorization decision.** The browser can alter it. Derive coverage and allowed evidence IDs server-side from a trusted bundled source. [VERIFIED: 03-CONTEXT.md]
2. **Valid JSON treated as grounded output.** Structured Outputs guarantees shape, not whether an ID belongs to this packet or whether each rendered claim has a chip; application validation remains mandatory. OpenAI documents schema adherence, while the locked contract requires local ID validation. [CITED: https://developers.openai.com/api/docs/guides/structured-outputs] [VERIFIED: 03-CONTEXT.md]
3. **Failing over after invalid output.** That changes a safety/grounding failure into a second model attempt, prohibited by D-04. Return insufficient evidence instead. [VERIFIED: 03-CONTEXT.md]
4. **Accidentally treating a 429 quota/billing response as a permanent configuration failure.** OpenAI documents that 429 can also mean quota exhaustion; preserve only a compact provider/error-type diagnostic and continue according to the locked rate-limit policy. [CITED: https://developers.openai.com/api/docs/guides/error-codes] [VERIFIED: 03-CONTEXT.md]
5. **Allowing `Find more sources` to mutate scope.** In this phase it is explanatory UI only; no fetch, importer invocation, reporting substitution, or seed mutation. [VERIFIED: 03-CONTEXT.md]
6. **Labelling a deterministic answer as live.** Keep `Bundled demo answer` distinct from the active provider label. [VERIFIED: 03-CONTEXT.md]
7. **Losing accessibility by making citation chips only visual.** Make them buttons with source/locator labels and route them through the existing evidence detail state. [VERIFIED: src/App.tsx, 03-CONTEXT.md]

## Code Examples

```ts
const allowed = new Map(packet.map((record) => [record.id, record]));
const parsed = parseGroundedAnswer(providerPayload);

if (parsed.status === "answered" && parsed.blocks.every((block) =>
  block.text.trim() && block.evidenceIds.length && block.evidenceIds.every((id) => allowed.has(id)),
)) return parsed;

return { status: "insufficient", blocks: [] };
```

The schema should set `additionalProperties: false`, make all object fields required, bound answer text with the provider’s output-token cap, and expose `evidenceIds` as strings rather than URLs. OpenAI’s strict JSON Schema mode is the right output mechanism for GPT-5.6. [CITED: https://developers.openai.com/api/docs/guides/structured-outputs]

```ts
for (const provider of configuredProvidersInLockedOrder) {
  const result = await provider.answer(question, packet);
  if (result.kind === "success") return validateAgainstPacket(result.answer, packet);
  if (result.kind !== "transient_failure") return insufficientEvidence();
}
return providerUnavailable(lastFailure);
```

Validate the response before returning from the server. Do not place this fallback loop in React. [VERIFIED: 03-CONTEXT.md]
