# Phase 04.1: Local Account Settings — Context

## Phase boundary

Add a fourth Account tab that consolidates existing local controls and makes their privacy boundary legible. This is a local settings surface, not authentication or a user-profile system.

## Locked decisions

- The tab is named **Account** for familiar navigation, but has no sign-in, account creation, sync, cloud persistence, or server profile.
- Reuse existing selected coverage, Curated topic, Curated reset, and Chat-thread state rather than creating a preference service.
- Account may change coverage, manage the same local Curated topics, reset Curated personalization, and explicitly clear local Chat history.
- Curated reset preserves Chat history. Clearing Chat history is a separate native-confirm action with clear copy.
- Show a concise local-only privacy explanation and provider availability/status only. Never expose API keys, provider configuration inputs, model controls, raw local-storage payloads, activity history, or political-profile inferences.
- Do not add full-document PDF/transcript RAG, source expansion/search, retrieval changes, provider memory, accounts, auth, databases, APIs, or dependencies.
- Reuse the existing bottom-nav, buttons, confirmation, status, and responsive styling patterns.

## Verification focus

- Account is keyboard accessible and has clear active navigation.
- Changing coverage affects the existing app selection without widening configured coverage.
- Topic changes and Curated reset use the existing isolated Curated state.
- Chat clear is explicit and does not alter Curated state; Curated reset does not alter Chat history.
- No secret appears in browser code or Account UI.
