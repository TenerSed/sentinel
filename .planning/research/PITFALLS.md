# Lamplighter Pitfalls

Research date: 2026-07-18. Phase labels are suggested because no roadmap exists yet.

| Pitfall | Warning signs | Prevention | Phase | Confidence |
|---|---|---|---|---|
| **A source is official-looking but does not support the displayed claim.** Political claims can be summaries, proposals, or quotations stripped of their status, jurisdiction, date, and speaker. | A card cannot point to an exact source passage; the model returns a citation ID it was not given; a headline says “passed” while the primary record says “introduced.” | Store claim-to-evidence links, not just source URLs: source version, publisher, retrieved time, quoted/extracted span, jurisdiction, content type, and locator. Render a claim only when every factual sentence has an allowed stored citation; label summaries and distinguish introduced/passed/signed. Fail closed to “I can’t verify that from the available sources.” | 1. Source/evidence schema; 3. grounded Chat | High — project constraint plus [NARA quality guidance](https://www.archives.gov/files/records-mgmt/policy/digitization-quality-mgmt-guide.pdf) emphasizes authenticity, accuracy, usability, metadata, and validation. |
| **Stale records are presented as current.** Government pages, RSS items, office-holder rosters, and bill statuses change at different rates; a feed date is not an effective-date or last-action date. | Missing `retrieved_at`; the same item reappears as new; “current representative” conflicts with the cited roster; feeds keep serving an old cached body. | Keep `published_at`, `updated_at`, `retrieved_at`, `effective_at`, canonical URL, content hash, and supersedes/retracted state. Give source types explicit freshness windows; show “as of” dates; refresh roster/status sources before demo; never infer current status from publication date alone. | 1. Source registry; 2. ingestion | High — Next.js GET routes can be cached only when opted in, so accidental cache policy can preserve stale results; see [Route Handlers](https://nextjs.org/docs/app/getting-started/route-handlers). |
| **RSS/news ingestion silently loses, duplicates, or misattributes records.** Feeds are truncated, HTML-heavy, backfilled, reordered, or reuse GUIDs; news is useful context but not primary proof of a government action. | Empty descriptions become claims; multiple cards have same canonical URL; item dates are blank/future; links redirect to unrelated content; a news story is the only support for a legal/status claim. | Treat RSS as discovery. Fetch and preserve the linked canonical page when permitted; normalize URLs, hash normalized content, and dedupe by canonical URL + hash. Require a primary government record for legislative/status assertions; retain publisher/type labels for secondary reporting. Quarantine failed/changed parses rather than inventing fields. | 2. ingestion | High — [NARA’s digitization strategy](https://www.archives.gov/digitization/strategy.html) treats descriptive/technical metadata and quality control as necessary for usable, authentic digital records. |
| **PDF/HTML extraction fabricates clean text from poor source material.** Scanned agendas, tables, amendments, headers, multi-column PDFs, and OCR errors can invert names, numbers, or vote results. | Extracted text is tiny/garbled; page count or file hash absent; cited page does not contain the asserted wording; text differs visibly from the PDF. | Preserve original bytes and SHA-256; record extractor/version and page spans. Validate page count and minimum text coverage; OCR only when needed; use page-image/manual spot checks for demo sources. Do not answer a precise claim from an unverified low-quality extraction; cite the original page. | 2. ingestion; 3. evidence UI | High — [NARA PDF guidance](https://www.archives.gov/files/records-mgmt/initiatives/pdf-records.html) says OCR text must be identical in content and appearance to its source and rejects processes that alter/degrade it. |
| **Video transcripts attribute words to the wrong person or timestamp.** Captions may be auto-generated, delayed, edited, incomplete, or cover several speakers; timestamps often point near rather than at the claim. | Transcript has no provenance/language/caption mode; speaker labels are model-generated; a timestamp opens before/after the quoted exchange; video has been removed or replaced. | Store video ID, channel/owner, duration, retrieved time, transcript provider/caption type, language, text segments, and start/end seconds. Use exact segment citations and quote only short verified spans; label unknown speakers as unknown; require a human playback check for every demo video claim. Link to the video at the stored time, not a transcript-only page. | 2. ingestion; 3. evidence UI | Medium — [YouTube’s terms](https://www.youtube.com/t/terms) make content the provider’s responsibility and terms can change; use the government owner’s original publication where possible and verify third-party transcript use/licensing. |
| **Citation rendering looks trustworthy but is not resolvable.** Link rot, redirects, source revision, missing page/timestamp, or a citation attached to the whole answer hides which evidence supports which sentence. | “Source” links open a homepage; document citations lack page number; citations cannot be clicked from Chat; answer has more factual claims than evidence links. | Validate on ingest and before seed release: HTTPS URL, final URL/domain policy, HTTP success, source hash/version, and locator that resolves. Model output should be structured as answer claims plus source IDs/locators; server validates IDs belong to retrieved records and discards unsupported claims. Test every seeded citation in CI/a small script. | 1. evidence schema; 3. grounded Chat | High — OpenAI recommends pinned versions and evals for consistent output in production; see [API backward compatibility guidance](https://platform.openai.com/docs/api-reference/backward-compatibility). |
| **Personalization becomes political profiling or leaks between sessions.** Reading and question history can reveal ideology, issue salience, or affiliations even without accounts. Shared caching can show one visitor’s curated results to another. | Raw queries/history are retained indefinitely; client sends history to third parties by default; a shared response cache varies by no user/session key; users cannot reset profile. | Default to local, pseudonymous, short-lived preference signals (topic/jurisdiction/content type), not inferred ideology. Provide visible “why this” and reset controls; do not store raw chat by default; keep API calls server-side and send minimum context. Do not put personalized output in shared cache; use request-scoped data. | 4. personalization | High — Next.js warns shared `use cache` cannot directly access cookies/headers and documents separate private caching; see [use cache](https://nextjs.org/docs/app/api-reference/directives/use-cache). |
| **SQLite works locally but fails or diverges in deployment.** `better-sqlite3` needs a writable persistent filesystem and is unsuitable for multiple independent serverless instances. Bundled seed DBs may be read-only or changes vanish on restart. | `SQLITE_READONLY`/`database is locked`; uploaded deployment has no durable writes; two instances return different feeds; production path differs from local path. | Make the judged path a single Node process with a bundled read-only seed DB and explicit write location only if live ingestion is enabled. Disable live writes on ephemeral/serverless hosting, or deploy a persistent single instance. Do not promise multi-instance SQLite; WAL does not make it distributed. | 1. deployment foundation; 5. demo hardening | High — Next.js says a single `next start` Node process supports all features, while multi-instance deployments need shared cache/tag coordination; see [deployment guidance](https://nextjs.org/docs/app/guides/deploying-to-platforms). |
| **A live dependency breaks the demo.** RSS endpoints, video hosts, model calls, or API keys fail/rate-limit while judges evaluate; dynamically fetched pages can drift after recording. | App needs an API key to open; initial feed has loading/error state; demo data varies between runs; ingestion runs during page load. | Seed all demo screens, citations, transcripts, and answer fixtures into SQLite; default to deterministic offline/demo mode with live ingestion as an explicit server-only action. Add one smoke command that starts the app and verifies the core Recent → evidence → Chat path without a key. Record against the same seed build. | 5. demo hardening | High — [Devpost’s rules](https://openai.devpost.com/rules) require a runnable project and a clear audio demo under three minutes; reliability matters more than live breadth. |
| **OpenAI output escapes the evidence boundary.** Prompt instructions alone do not prevent unsupported assertions, prompt injection inside public documents, changed model behavior, or malformed structured output. | The model cites arbitrary URLs; source text says “ignore prior instructions”; answer includes claims with no source IDs; a model/model-version change shifts output quality. | Delimit source text as untrusted data; deterministic server retrieval chooses records; request a strict structured schema with only provided source IDs/locators; validate it server-side and omit invalid claims. Pin the intended model version where available, log request IDs, and keep a small adversarial/evidence evaluation set. | 3. grounded Chat | High — the [OpenAI API reference](https://platform.openai.com/docs/api-reference/backward-compatibility) recommends pinned versions, evals, and request-ID logging; use the [official Node SDK](https://platform.openai.com/docs/quickstart/make-your-first-api-request) only server-side. |
| **Submission violates or misses Build Week requirements.** A polished demo can still be ineligible if the video/repository/docs/usage evidence are missing, third-party data lacks permission, or pre-existing work is indistinguishable from new work. | No dated commits; README has no one-command path/sample data/license; video is >3 minutes or lacks audio/model explanation; repo is private but not shared; copyrighted media appears in video. | Maintain dated commits and a short build log; MIT license; README with one-command local path, demo mode, data provenance, and Codex/GPT-5.6 use. Before submit, check the video is public YouTube, under three minutes, voiced, and demonstrates the app plus tool usage; verify all source/API/video use terms. | 5. submission | High — [official rules](https://openai.devpost.com/rules) and [FAQ](https://openai.devpost.com/details/faqs) require a runnable project, repo access, public YouTube demo with audio, README/sample-data guidance, and evidence of Codex/GPT-5.6 work during the submission period. |

## Minimum release gates

1. Every visible factual claim has one or more stored, clickable evidence records with an exact page or timestamp when applicable.
2. Every seeded source, video timestamp, document page, and demo Chat answer is verified once immediately before recording.
3. Demo mode starts and completes the primary flow with network and OpenAI credentials absent.
